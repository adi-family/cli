# Internationalization

## Why This Rule Exists

ADI supports 9 languages. Every user-facing string must go through the translation system so non-English users get a localized experience. Hardcoded English strings are bugs.

## Rules

### 1. All User-Facing Strings Use `t!()`

```rust
// BAD
out_info!("Plugin installed successfully");

// GOOD
out_info!("{}", t!("plugin-install-success"));
```

### 2. Key Naming Convention

Format: `domain-action-detail`

```ftl
# GOOD
auth-login-success = Logged in successfully
plugin-install-error = Failed to install plugin { $name }
task-create-confirm = Create task "{ $title }"?

# BAD
success = Success
msg1 = Logged in
error_plugin = Failed
```

### 3. English First

Always write `en-US/messages.ftl` first. It serves as the fallback for missing translations in other locales.

### 4. Variables Use Fluent Syntax

```ftl
greeting = Hello, { $name }!
item-count = { $count ->
    [one] { $count } item
   *[other] { $count } items
}
```

### 5. Log Messages Are Exempt

Internal debug/trace logs do not require translation. Only user-visible output (CLI messages, GUI labels, error messages shown to users) must be translated.

### 6. Supported Locales

`en-US`, `de-DE`, `es-ES`, `fr-FR`, `ja-JP`, `ko-KR`, `ru-RU`, `uk-UA`, `zh-CN`

## The Test

*"Would a non-English user see raw English from this code path?"* If yes, wrap it in `t!()`.
